# Versioning and Git Workflow

## 提交规则

- 每次有效修改单独提交，不把无关改动混在同一提交中。
- 提交信息应直接说明修改内容，例如：
  - `Fix empty Sentinel-1 quarterly composites`
  - `Export water samples for visual QA`
- 每次代码提交同步更新 `HISTORY_LOG.md`。
- 临时日志、错误输出、导出影像和本地缓存不提交到仓库。
- 已发布脚本不直接删除或用不兼容代码覆盖；重大方法变化应新建脚本。

## 分支规则

- `main`：仅保存已完成基本静态检查、可交付用户运行的版本。
- 实验性修改使用独立分支，命名格式：
  - `experiment/<简短主题>`
  - `fix/<简短问题>`
  - `feature/<简短功能>`
- 实验分支至少完成语法检查和关键逻辑检查后才能合并到 `main`。
- 合并后保留提交历史，便于追踪实验过程。

## 版本标签

采用语义化版本：

- `v0.1.0`：基础随机森林土地利用分类版本。
- `v0.2.0`：增强版分层随机森林版本。
- `v0.x.y`：研究和验证阶段的兼容性改进或问题修复。
- `v1.0.0`：经过人工样本核查、独立精度验证且满足最终验收要求的稳定版本。

禁止移动或重复使用已经推送的标签。版本内容发生变化时创建新标签。

## GitHub Release

重要稳定版本创建 GitHub Release。Release 至少包括：

- 版本目的和主要变化；
- 对应主脚本；
- 输入资产和数据时间范围；
- 核心参数与类别编码；
- 训练、调参和测试设计；
- 精度结果及其适用边界；
- 输出文件；
- 已知问题和复现步骤。

普通修复提交不单独创建 Release，在下一个稳定版本中汇总。

## 验证要求

提交到 `main` 前至少完成：

- JavaScript 语法检查；
- GEE资产路径和波段名检查；
- 类别编码、训练样本和空间划分检查；
- 导出文件名和输出目录检查；
- 无法在本地完成的 GEE 服务端运行验证必须明确记录。
